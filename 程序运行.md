## 一、批处理
> **RISC-V 寄存器编号和别名**
> RISC-V 寄存器编号从`0~31`，表示为`x0~x31`。其中：
>
> * `x10~x17`：对应`a0~a7`
> * `x1`：对应`ra`

#### 1. 内存布局🖼️

首先在`user/src/linker.ld`文件中编写好链接脚本，再通过`user/.cargo/config`去声明使用链接脚本

1. `linker.ld`

   ```assembly
   OUTPUT_ARCH(riscv)
   ENTRY(_start)
   
   BASE_ADDRESS = 0x80400000;
   
   SECTIONS
   {
       . = BASE_ADDRESS;
       .text : {
           *(.text.entry)
           *(.text .text.*)
       }
       .rodata : {
           *(.rodata .rodata.*)
           *(.srodata .srodata.*)
       }
       .data : {
           *(.data .data.*)
           *(.sdata .sdata.*)
       }
       .bss : {
           start_bss = .;
           *(.bss .bss.*)
           *(.sbss .sbss.*)
           end_bss = .;
       }
       /DISCARD/ : {
           *(.eh_frame)
           *(.debug*)
       }
   }
   ```
2. `config`

   ```toml
   [build]
   target = "riscv64gc-unknown-none-elf"
   
   [target.riscv64gc-unknown-none-elf]
   rustflags = [
       "-Clink-args=-Tsrc/linker.ld", "-Cforce-frame-pointers=yes"
   ]
   
   ```

#### 2. 系统调用🍔

+ 确定系统调用有哪些
    ```rust
    /// 功能：将内存中缓冲区中的数据写入文件。
    /// 参数：`fd` 表示待写入文件的文件描述符；
    ///      `buf` 表示内存中缓冲区的起始地址；
    ///      `len` 表示内存中缓冲区的长度。
    /// 返回值：返回成功写入的长度。
    /// syscall ID：64
    fn sys_write(fd: usize, buf: *const u8, len: usize) -> isize;
    
    /// 功能：退出应用程序并将返回值告知批处理系统。
    /// 参数：`exit_code` 表示应用程序的返回值。
    /// 返回值：该系统调用不应该返回。
    /// syscall ID：93
    fn sys_exit(exit_code: usize) -> !;
    ```

+ 系统调用的层级关系
    ```rust
    fn program1() {//-bin-应用程序
        pub fn write_str() {//-console.rs-具体功能
            pub fn write() {//-lib.rs-系统功能
                pub fn sys_write() {//-syscall.rs-系统调用
                    pub fn syscall() {//-syscall.rs-系统调用
                        "ecall"; //汇编
                    }
                }
            }
        }
    }
    ```

#### 1. 将程序连接到内核🚀️

+ 编译❗：

  + `Makefile` 文件中编写了编译的全部过程，其中强调的是，编译过程调用了 `linker-qemu.ld` ，进行了内存布局的设置。[^1]

  ```makefile
  # Building mode argument
  ifeq ($(MODE), release)
  	MODE_ARG := --release
  endif
  
  # BOARD
  BOARD := qemu
  
  kernel:
  	@cd ../user && make build
  	@echo Platform: $(BOARD)
  	@cp src/linker-$(BOARD).ld src/linker.ld
  	@cargo build $(MODE_ARG)
  	@rm src/linker.ld
  ```

  + 这里声明了内存空间（`.text/.rodata(只读数据)/.data(已初始化数据)/.bss(未初始化数据 + 栈！)`），注意这里**并没有**给出来栈空间分配，这里由代码段中的`.text.entry`程序来完成，而`.text.entry`里面的代码又是由`entry.asm`中指出的。（这里不明白这段汇编代码含义的，[查看连接](./内存分配.md)）

  ```assembly
  OUTPUT_ARCH(riscv)
  ENTRY(_start)
  BASE_ADDRESS = 0x80200000;
  
  SECTIONS
  {
      . = BASE_ADDRESS;
      skernel = .;
  
      stext = .;
      .text : {
          *(.text.entry)
          *(.text .text.*)
      }
  
      . = ALIGN(4K);
      etext = .;
      srodata = .;
      .rodata : {
          *(.rodata .rodata.*)
          *(.srodata .srodata.*)
      }
  
      . = ALIGN(4K);
      erodata = .;
      sdata = .;
      .data : {
          *(.data .data.*)
          *(.sdata .sdata.*)
      }
  
      . = ALIGN(4K);
      edata = .;
      .bss : {
          *(.bss.stack)
          sbss = .;
          *(.bss .bss.*)
          *(.sbss .sbss.*)
      }
  
      . = ALIGN(4K);
      ebss = .;
      ekernel = .;
  
      /DISCARD/ : {
          *(.eh_frame)
      }
  }
  ```

- `link_app.S`：

  + 在Rust项目中，如果存在 `build.rs` 文件，Cargo会在编译主项目之前自动编译并执行它，`build.rs` 会编译产生 `link_app.S` 文件
  + `link_app.S`文件就是为 user 中 **应用程序** 分配了存放的空间

  ```assembly
   .align 3
      .section .data
      .global _num_app
  _num_app:
      .quad 5
      .quad app_0_start
      .quad app_1_start
      .quad app_2_start
      .quad app_3_start
      .quad app_4_start
      .quad app_4_end
  
      .section .data
      .global app_0_start
      .global app_0_end
  app_0_start:
      .incbin "../user/target/riscv64gc-unknown-none-elf/release/00hello_world.bin"
  app_0_end:
  
  ...
  
      .section .data
      .global app_4_start
      .global app_4_end
  app_4_start:
      .incbin "../user/target/riscv64gc-unknown-none-elf/release/04priv_csr.bin"
  app_4_end:
  ```

- `entry.asm`

  + 在接下来的过程中，我们将会通过**用户栈**和**内核栈**实现由用户态**Trap到**内核态，以及由内核态**返回**用户态，所以我们需要栈空间，栈空间的声明则是在这里完成的，栈空间是 `.bss` 内存段中的一“小”段。

  ```assembly
      .section .text.entry
      .globl _start
  _start:
      la sp, boot_stack_top 		;将 boot_stack_top 内存地址 加载到 sp寄存器中
      call rust_main 				;定义在 main.rs 中，是一个函数;
  								;call 用于指定程序入口，整个代码是从 内存地址 的第一行开始执行，也就是.text.entry这段的第一行开始的
  
      .section .bss.stack
      
      .globl boot_stack_lower_bound
  boot_stack_lower_bound:
  
      .space 4096 * 16
      
      .globl boot_stack_top
  boot_stack_top:
  ```

  ```shell
  +---------------------+ boot_stack_top (初始 sp 指向这里，sp 向着低地址方向增长，也就是减小，在图中向下移动)
  |                     | 
  |     栈空间           | 大小为 16KB (4096 * 16)
  |    (向下增长)        |
  |                     |
  +---------------------+ boot_stack_lower_bound
  |                     | 
  |     其他内存区域      | （如 .text, .data 等）
  |                     | 
  +---------------------+ 0x80200000 (BASE_ADDRESS)
  ```

#### 2. 内核编译[^2]🦉

1. 首先在 `main.rs` 中我们需要导入所有的 **模块**
   ```rust
   #[macro_use]
   mod console;
   pub mod batch;
   mod lang_items;
   mod logging;
   mod sbi;
   mod sync;
   pub mod syscall;
   pub mod trap;
   ```

2. 然后在 `main.rs` 中导入全局汇编 `entry.asm` 和 `link_app.S`，并清空 `.bss(未初始化 + 栈)` 段中的内容，因为是未初始化的值，默认为 0
   ```rust
   use core::arch::global_asm;
   
   global_asm!(include_str!("entry.asm"));
   global_asm!(include_str!("link_app.S"));
   
   /// clear BSS segment
   fn clear_bss() {
       unsafe extern "C" {
           safe fn sbss();
           safe fn ebss();
       }
       unsafe {
           core::slice::from_raw_parts_mut(sbss as usize as *mut u8, ebss as usize - sbss as usize)
               .fill(0);
       }
   }
   ```

3. 万事俱备，只欠 `main函数` 了🐦‍🔥，这里的 `main函数` 名为 `fn rust_main()`，这个程序入口是由 `entry.asm` 指定的
   ```rust
   /// os
   #[unsafe(no_mangle)] 			// 防止编译器管理过程中，rustc器对函数名进行混淆，确保链接时能找到 rust_main
   pub fn rust_main() -> ! {
       unsafe extern "C" { 		// 获得 link-qemu.ld 中分配好的内存地址
           safe fn stext(); 				
           safe fn etext(); 				
           safe fn srodata(); 					
           safe fn erodata(); 				
           safe fn sdata(); 					
           safe fn edata(); 					
           safe fn sbss(); 					
           safe fn ebss(); 					
           safe fn boot_stack_lower_bound(); 	
           safe fn boot_stack_top(); 			
       }
       clear_bss();				// 调用clear_bss()函数（上文刚刚讲过，忘记挨打！），清空 .bss 段
       
       // 省略 日志初始化
       // 省略 打印内存布局信息
       
       trap::init();				// 初始化 中断
       batch::init();				// 初始化 批处理
       batch::run_next_app();		// 执行下一个程序（也就是第一个程序）
   }
   ```

[^1]:为什么在os中使用Makefile在编译阶段运行linker-qemu.ld来分配内存空间，而对于user用户程序则反而是在.cargo/config.toml中的代码指出使用linker.ld来分配内存？
[^2]:编译过程，其实主要和Cargo构建项目的顺序一致，首先Cargo会先读取Cargo.toml、.cargo/config.toml（这里会覆盖之前的Cargo.toml）、build.rs（如果有的话）；然后根据.toml配置文件导入加载依赖，且编译并**执行**build.rs；然后就开始编译内核代码喽，入口文件一般是src/main.rs或者是src/lib.rs（这个由Cargo.toml指定），Rust编译器（rustc）会将main.rs中的每个模块编译成为LLVM IR，优化，再编译为机器码；最后作链接，Cargo 调用 rust-lld生成可执行文件。
[^3]:个人认为，在执行之前，便已经编译好了全部二进制文件，所以程序可以知道从.text.entry的第一行开始执行，然后再call调用rust_main()。



# 补充插页

```rust
/**
* 批处理
* src/main.rs 完整代码
*/
#![deny(missing_docs)]
#![deny(warnings)]
#![no_std]
#![no_main]

use core::arch::global_asm;

use log::*;
#[macro_use]
mod console;
pub mod batch;
mod lang_items;
mod logging;
mod sbi;
mod sync;
pub mod syscall;
pub mod trap;

global_asm!(include_str!("entry.asm"));
global_asm!(include_str!("link_app.S"));

/// clear BSS segment
fn clear_bss() {
    unsafe extern "C" {
        safe fn sbss();
        safe fn ebss();
    }
    unsafe {
        core::slice::from_raw_parts_mut(sbss as usize as *mut u8, ebss as usize - sbss as usize)
            .fill(0);
    }
}

/// the rust entry-point of os
#[unsafe(no_mangle)]
pub fn rust_main() -> ! {
    unsafe extern "C" {
        safe fn stext(); // begin addr of text segment
        safe fn etext(); // end addr of text segment
        safe fn srodata(); // start addr of Read-Only data segment
        safe fn erodata(); // end addr of Read-Only data ssegment
        safe fn sdata(); // start addr of data segment
        safe fn edata(); // end addr of data segment
        safe fn sbss(); // start addr of BSS segment
        safe fn ebss(); // end addr of BSS segment
        safe fn boot_stack_lower_bound(); // stack lower bound
        safe fn boot_stack_top(); // stack top
    }
    clear_bss();
    logging::init();
    println!("[kernel] Hello, world!");
    trace!(
        "[kernel] .text [{:#x}, {:#x})",
        stext as usize, etext as usize
    );
    debug!(
        "[kernel] .rodata [{:#x}, {:#x})",
        srodata as usize, erodata as usize
    );
    info!(
        "[kernel] .data [{:#x}, {:#x})",
        sdata as usize, edata as usize
    );
    warn!(
        "[kernel] boot_stack top=bottom={:#x}, lower_bound={:#x}",
        boot_stack_top as usize, boot_stack_lower_bound as usize
    );
    error!("[kernel] .bss [{:#x}, {:#x})", sbss as usize, ebss as usize);
    trap::init();
    batch::init();
    batch::run_next_app();
}
```
